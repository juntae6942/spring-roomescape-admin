### 1. 막힌 것의 종류
이번에 막힌 것은 어떤 종류의 어려움이었는가? (해당하는 것에 체크)
- [ ] 개념 자체를 모르겠다 (예: "스프링 빈이 뭔지 모르겠다")
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다 (예: "JdbcTemplate 문법을 모르겠다")
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다 (예: "계층 분리를 이렇게 해도 되나?")
- [X] 기타: 테스트 코드의 의도를 파악하지 못하여 막혔습니다.
아래의 요구사항을 기반으로 코드를 작성하였는데, 테스트 코드에서는 reservations만을 호출하는 테스트로 작성되어 있어서,
ReservationTime을 저장하는 API 호출이 먼저 진행되어야 할 것 같은데 그 부분이 없어서 해당 timeId의 time을 Reservation을 저장하는 과정에 포함해야 하는가?
고민했습니다.
```
Reservation 클래스의 time 필드를 String → ReservationTime 객체로 변경
예약 추가 요청 본문: time → timeId
예약 조회 응답: time을 객체로 ({id, startAt})
```
하지만 AutoIncrement로 지정된 schema.sql이 주어져서 이 또한 올바른 방향이 아니라고 생각했습니다.
그래서 브라운에게 질문을 통해 테스트 코드를 수정하는 방식으로 해결하였습니다.

### 2. 이번 타임의 학습 전략
- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
공식문서 중에 예시코드가 있는 부분을 기점으로 그 주위의 부분들을 읽어서 학습하는 i+1 학습법을 실행했습니다.
- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.
시간 관리를 추가하면서 Reservation에 수정이 필요했고, BeanPropertySqlParameterSource부분에서 문제가 발생했습니다. 이 문제를 해결하기 위해서 다시 MapSqlParameterSource를 사용하였습니다.
하지만 이 문제를 알아보고 싶어서 공식문서를 찾아보았지만, 찾기 어려웠습니다. 그래서 AI를 활용해서 문제의 원인을 파악하기 위해서 시도했습니다.
그 결과, BeanPropertySqlParameterSource은 Getter 메서드 이름을 기반으로 파라미터를 추출하는 아주 단순한 도구라는 것을 알았고, 이러한 불편함 때문에 JPA가 발전됐다는 것을 알았습니다.

### 4. AI 피드백
- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고,
  유용했던 제안 1가지 이상 기록  
  제안하는 학습 전략: 'AI 인덱서'를 활용한 가설 검증 핑퐁 (Hypothesis-Driven Ping-pong)  
피드백 내용: 공식 문서는 개별 기술의 '명세서'이므로, 자바 객체의 중첩 구조와 데이터베이스의 참조 무결성이 얽힌 복합적인 문제 상황에서 곧바로 해답을 찾기에는 비효율적일 수밖에 없습니다. 
공식 문서의 방대함이라는 단점을 극복하기 위해, AI를 '지식의 정답 자판기'가 아니라 '공식 문서의 똑똑한 인덱서(색인기)'로 활용하는 전략을 제안합니다.
문제에 부딪혔을 때 곧바로 방대한 문서를 뒤지거나 AI에게 막연히 원인을 묻지 말고, "객체 안에 객체가 있는 구조인데, 이 클래스가 1뎁스 이상의 깊은 탐색을 지원하지 않는 게 원인일까?"처럼 본인의 가설을 먼저 AI에게 지시(Tell)하듯 던지세요. 
외부 객체의 상태를 함부로 묻지 않고 스스로 판단하여 행동하게 만드는 객체지향의 원칙처럼, 
AI에게도 내 가설을 먼저 전달하여 문서의 어느 부분을(어떤 키워드를) 봐야 할지 좁혀낸 뒤에 공식 문서의 코드를 확인하는 방식으로 피드백 루프를 개선해 보세요.
